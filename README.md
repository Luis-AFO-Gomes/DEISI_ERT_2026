# DEISI_ERT_2026
Example project for ERT@ULHT, academic year 2026

Vibecoding using requirements and tests (BDD - Behaviour Driven Development)

## Working Basis
The project development follows the ERT course lab classes, available on GitBook in **[Requirements Engineering & Testing — Lab Classes](https://ulht-jcb.gitbook.io/requirements-engineering-and-testing)**

---

## Introduction

This project is a practical example designed to demonstrate how to test software at different levels and using different methodologies:

- **Unit Testing** — verifying individual components in isolation
- **TDD (Test-Driven Development)** — writing tests before writing production code, letting tests drive design
- **BDD (Behaviour-Driven Development)** — expressing expected system behaviour in plain language, bridging technical and business perspectives

The goal is to provide a concrete, working example that can be followed alongside the lab classes, showing how requirements evolve into tests and how tests validate the implemented software.

## Project Structure

```
DEISI_ERT_2026/
├── bdd/            # BDD feature files and step definitions (Behave/Gherkin)
├── docs/           # Project documentation: vision, requirements, use cases, test cases
├── sql/            # SQL scripts for the database
├── src/
│   ├── desktop_app/            # Main application package
│   │   ├── config/             # Runtime configuration files (including DB connection)
│   │   ├── dao/                # Data Access Objects (database operations)
│   │   ├── models/             # Domain models/entities
│   │   └── utils/              # Utility and helper modules
│   └── DEISI_ERT_2026.egg-info/ # Packaging metadata generated by installation/build tools
├── templates/      # UI or report templates
└── tests/          # Unit and TDD tests (pytest and unittest)
```

- **bdd/** — Contains `.feature` files written in Gherkin and the corresponding Python step definitions used by Behave to run acceptance/BDD tests. See [bdd/README.md](bdd/README.md) for instructions on how to run them.
- **docs/** — All project documentation artefacts: vision, objectives, requirements, use cases, acceptance criteria, test cases, and traceability.
- **sql/** — SQL scripts to set up and populate the project database.
- **src/** — The application source code. Inside `src/desktop_app`, the code is organised into configuration, data access objects (DAO), domain models, and utility modules.
- **templates/** — Templates used by the application.
- **tests/** — Unit tests and TDD tests written with pytest and unittest. See [tests/README.md](tests/README.md) for instructions on how to run them.

## Install and Run the Application (CRUD Livro)

The code is runnable and includes a functional CRUD flow for the `Livro` class, with data read from a Microsoft SQL Server (MSSQL) database.

### Environment setup

1. Clone the repository:

```bash
git clone https://github.com/Luis-AFO-Gomes/DEISI_ERT_2026.git
cd DEISI_ERT_2026
```

2. Create and activate a Python virtual environment:

```bash
python -m venv .venv
```

Windows (PowerShell):

```bash
.venv\Scripts\Activate.ps1
```

3. Install project dependencies:

```bash
python -m pip install -e .
```

### Database setup (MSSQL)

1. Ensure a running MSSQL instance is available.
2. Execute the SQL script [sql/biblioteca.sql](sql/biblioteca.sql) to create and seed the `biblioteca` database.
3. Review and update [src/desktop_app/config/config.ini](src/desktop_app/config/config.ini) with your local database settings.

The application reads database connection properties from [src/desktop_app/config/config.ini](src/desktop_app/config/config.ini) (`driver`, `host`, `db`, `user`, `pass`, encryption options). These values should be adapted to each local MSSQL installation.

### Run the application

From the project root, run:

```bash
python -m desktop_app.main
```

Alternative (after installation):

```bash
library-manager
```

Students are invited to grow the application by adding classes or other functionalities. This extension work is expected and will be required for error correction activities.

## Conclusion

The tests and source code in this project contain **intentional errors**. These errors are deliberate and serve as learning material — they are meant to be discovered when running the tests, reading test results, and practising debugging. Finding and understanding these errors is part of the learning experience.
